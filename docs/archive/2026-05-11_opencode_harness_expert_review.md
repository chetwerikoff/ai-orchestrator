## 2. High-level architecture

The intended split of responsibilities is:

```text
Claude
  = task planner / business-logic architect / final domain reviewer

ai-git-orchestrator
  = workflow harness / dispatcher / context builder / prompt builder / test runner / review coordinator

OpenCode
  = execution harness for coding agents
  = reads files, edits files, runs terminal commands, applies changes, works with repo context

Local Qwen
  = cheap local coding/fixing/review model connected through OpenCode

Codex
  = code reviewer / implementation correctness reviewer

Domain checks
  = deterministic Python scripts that verify H2N-specific behavior
```

Important principle:

```text
OpenCode replaces the need to build low-level file-editing and command-running tools.
It does NOT replace the higher-level workflow harness.
```

So I do not want to build a full custom coding agent from scratch. I want to build a controlled workflow around OpenCode.

## 5. File responsibilities

### 5.1 Task and planning files

#### `.ai-loop/context_bundle.md`

Generated by `context_builder.py` before Claude writes the task.

Contains:

```text
- current task request from user
- git status
- recent diff, if any
- project tree summary
- relevant files/snippets
- relevant tests
- previous failure output
- known domain notes
- previous final_status.md, if relevant
```

Purpose:

```text
Give Claude enough project context to write a precise task.
```

---

#### `.ai-loop/task.md`

Written by Claude task planner.

Should be structured:

```md
# Task

## Goal

## Background

## Relevant files

## Expected behavior

## Implementation plan

## Acceptance criteria

## Constraints

## Forbidden changes
```

Purpose:

```text
Universal, model-independent task specification.
```

It should not be tied specifically to OpenCode, Qwen, Cursor, Codex, or Claude.

---

#### `.ai-loop/agent_brief.md`

Written by Claude task planner.

More practical than `task.md`.

Contains:

```text
- where the coding agent should start
- likely files to inspect
- likely tests to run
- risky areas
- what not to touch
```

Purpose:

```text
Bridge between task planning and coding execution.
```

---

#### `.ai-loop/domain_risks.md`

Written by Claude task planner.

Contains H2N-specific risks, for example:

```text
- HUD is not the same as popup
- color_ranges are thresholds from config, not live player stats
- do not invent stat aliases
- do not change output schema unless explicitly required
- preserve raw extracted config data where possible
- school stats are reference groupings, not necessarily source-of-truth config values
```

Purpose:

```text
Force all downstream agents and reviewers to consider domain semantics.
```

## 6. Orchestrator source file responsibilities

## 8. Proposed workflow

```text
1. User describes desired change or idea.

2. Orchestrator runs context_builder.py.
   Output: .ai-loop/context_bundle.md

3. Claude planner reads context_bundle.md and writes:
   - .ai-loop/task.md
   - .ai-loop/agent_brief.md
   - .ai-loop/domain_risks.md

4. prompt_builder.py creates:
   - .ai-loop/opencode_qwen_prompt.md

5. opencode_runner.py launches OpenCode with local Qwen.
   Output:
   - code changes
   - .ai-loop/opencode_qwen_result.md
   - .ai-loop/opencode_session_log.md

6. Orchestrator captures:
   - git status
   - changed files
   - git diff

7. test_runner.py runs full and targeted tests.
   Output:
   - .ai-loop/test_output.txt
   - .ai-loop/targeted_test_output.txt

8. domain_gate.py runs H2N-specific deterministic checks.
   Output:
   - .ai-loop/domain_check_report.md

9. diff_guard.py checks safety.
   Output:
   - .ai-loop/diff_guard_report.md

10. Qwen cheap review checks obvious issues.
    Output:
    - .ai-loop/qwen_review.md

11. If tests fail or cheap review fails, Qwen fixer can attempt a small fix through OpenCode.

12. If local checks pass, Codex reviews code correctness.
    Output:
    - .ai-loop/codex_review.md

13. If Codex has no blocking issues, Claude reviews H2N business logic.
    Output:
    - .ai-loop/claude_business_review.md

14. If all gates pass:
    - .ai-loop/final_status.md

15. If any gate fails and can be retried:
    - .ai-loop/next_opencode_prompt.md

16. If the task exceeds local model capability:
    - .ai-loop/next_cursor_prompt.md
    - escalate to Cursor or Claude/Codex workflow
```

## 9. Safety rules to enforce in code, not only prompts

The orchestrator should stop or escalate if:

```text
- more than N files changed
- test files deleted
- output schema changed without explicit permission
- generated output files changed unexpectedly
- diff is too large
- pytest still fails after max retries
- domain_check reports regression
- Codex reports blocking issue
- Claude business review reports semantic failure
```

Suggested starting thresholds:

```text
max_changed_files_for_qwen = 5
max_patch_lines_for_qwen = 500
max_iterations = 5
max_qwen_fix_attempts = 2
```
