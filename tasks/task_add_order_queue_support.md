# User ASK

## Goal

Добавить параметр `order` при создании задачи: если он указан, задача должна попадать не в текущий auto-loop, а в папку/очередь `tasks`, формируя очередь задач для последующей обработки.

Задачи в очереди должны получать осмысленные имена файлов с номером в очереди, чтобы порядок выполнения был понятен и стабилен.

## Proposed approach (optional)

Добавить поддержку параметра `order` в сценарий/команду создания задачи.

Ожидаемое поведение:

1. Если `order` не указан:
   - сохранить текущее поведение;
   - задача создаётся так же, как сейчас, и попадает в auto-loop.

2. Если `order` указан:
   - задача НЕ должна автоматически попадать в `.ai-loop/task.md`;
   - задача должна быть сохранена в папку `tasks`;
   - файл задачи должен иметь номер очереди и осмысленное название.

Пример желаемого формата имени файла:

```text
tasks/001_short_meaningful_task_name.md
tasks/002_fix_dashboard_generation.md
tasks/003_add_order_queue_support.md
```

Точные правила именования должен определить planner, но важно:

- номер должен отражать порядок в очереди;
- название должно быть человекочитаемым;
- название должно быть безопасным для файловой системы;
- при конфликте номеров или имён поведение должно быть явно определено;
- существующие задачи не должны перезаписываться без явного намерения пользователя.

Нужно продумать, как именно `order` должен работать:

- `order` может быть явным номером, например `order=3`;
- или режимом постановки в очередь, где следующий номер определяется автоматически;
- если текущая архитектура уже содержит похожий механизм, использовать его, а не дублировать.

Planner должен проверить текущую архитектуру проекта и выбрать самый безопасный вариант.

## Constraints / context the planner may not know

- Новая логика не должна ломать текущий сценарий запуска auto-loop.
- Если `order` не указан, поведение должно остаться обратно совместимым.
- Задачи в `tasks` нужны как очередь задач, а не как немедленный запуск в loop.
- Названия задач должны быть осмысленными, а не просто timestamp/hash.
- Номера в очереди должны позволять быстро понять порядок выполнения.
- Нужно избегать случайного перетирания уже существующих задач.
- Если непонятно, какие конкретно файлы менять, planner должен сначала изучить структуру проекта и текущие scripts/docs, связанные с созданием задач.
